## 引言
本文件基于仓库中的 README.md、开发方案.md 以及 tiptap 集成说明.md，系统梳理项目“未知叙事 - 小说块编辑器”的未来发展方向。目标是在现有“块编辑 + Markdown + 双链”的基础上，逐步实现文件系统集成、多文档切换、搜索与大纲视图、协同编辑（yjs）、图谱视图（双链关系与反向链接），并进一步拓展角色图片差分管理、图文混排、富文本样式扩展、骰子投掷等高级功能。本文将给出阶段性目标、关键技术选型、实现思路与风险提示，帮助开发者与用户理解项目的长期愿景与落地节奏。

## 项目现状与技术栈
- 技术栈：Electron + React + TypeScript + Vite + Tailwind CSS + yjs + tiptap
- 当前实现：块编辑器已具备编辑态/渲染态切换、拖拽排序、Markdown 基础语法、导入导出、基础双链占位等能力；桌面端主进程与预加载脚本已就绪，具备扩展文件系统与协同编辑的基础。
- 开发方案：明确采用 tiptap 作为块编辑引擎，Lute（JS 版）负责 Markdown 解析渲染，localForage + IndexedDB 作为本地存储，Ant Design React + Tailwind CSS 提供 UI 基础组件与样式。

## 核心能力与当前实现
- 块数据模型：包含块 ID、类型、内容（Markdown 源码）、正向/反向双链引用、元数据（标签、创建/修改时间）等字段，满足双链与元数据扩展需求。
- 编辑器与渲染：BlockEditor 基于 tiptap，支持多种块类型与拖拽；MarkdownRenderer 提供基础渲染，预留双链占位。
- 数据管理：BlockManager 提供块的增删改查、拖拽重排、从 Markdown 导入、导出为 Markdown/JSON 等能力；useBlockManager 封装状态与操作。
- 桌面端：Electron 主进程负责窗口创建与安全策略，预加载脚本提供受限 IPC 接口，便于后续接入文件系统与对话框。

## 未来发展方向总览
围绕“本地优先 + 逐步云端扩展”的路线，分阶段推进以下方向：
- 短期（1-2 个月）：文件系统集成（打开/保存本地文件）、项目管理（多文档切换）、搜索与大纲视图、基础协同编辑（yjs）。
- 中期（2-4 个月）：图谱视图（双链关系与反向链接）、角色图片差分管理、图文混排、富文本样式扩展。
- 长期（4+ 个月）：骰子投掷、移动端适配（核心抽取 + 新项目）、云端同步与备份、插件化扩展。

## 详细规划与实现路径

### 1. 文件系统集成：打开/保存本地文件
- 目标：在桌面端提供“打开/保存”本地文件的能力，支持 .md/.json 等格式，结合 Electron 对话框与文件系统 API。
- 技术要点：
  - 在预加载脚本中暴露受控 IPC 方法（如 openFile、saveFile），在渲染进程中通过 window.electronAPI 调用。
  - 主进程使用 dialog.showOpenDialog/showSaveDialog，结合 fs 模块读写文件。
  - 与现有 BlockManager/导出导入流程对接，确保数据一致性。
- 风险与建议：
  - 注意安全：禁用 nodeIntegration，严格校验文件路径与内容。
  - 性能：大文档打开时可采用分块读取与懒加载。
  - 兼容：统一文件编码（UTF-8），处理异常文件格式。

### 2. 项目管理：多文档切换
- 目标：支持多个文档的创建、命名、列表管理与切换，提供侧边栏或标签页形式。
- 技术要点：
  - 在 App 层维护“当前文档 ID + 文档列表”的状态，使用 useBlockManager 返回的 blockManager 与导出/导入能力。
  - 通过 Electron 的菜单或窗口标题栏提供“新建/关闭/切换”操作。
  - 为每个文档建立独立的 IndexedDB 存储空间或命名空间，避免冲突。
- 风险与建议：
  - 切换时需持久化当前文档状态，避免丢失未保存更改。
  - 提供“未保存更改”提示与确认对话框。

### 3. 搜索功能与大纲视图：提升内容导航效率
- 目标：提供全文搜索与大纲视图，支持快速定位标题、段落与块。
- 技术要点：
  - 搜索：遍历块列表，基于 content 与 metadata 进行匹配；支持关键词高亮与跳转。
  - 大纲：解析块类型与层级（标题块），生成树形结构；点击跳转到对应块。
  - 与渲染态联动：点击大纲项时，编辑态自动切换到目标块并滚动定位。
- 风险与建议：
  - 大文档场景下，搜索与大纲生成需节流/防抖，避免卡顿。
  - 大纲需支持折叠/展开与层级缩进。

### 5. 图谱视图：展示双链关系与反向链接
- 目标：以可视化图谱呈现块之间的正向/反向链接，支持节点跳转与关系筛选。
- 技术要点：
  - 数据：从 BlockManager 获取 blocks，提取 references 与 referencedBy，构建邻接表。
  - 可视化：使用轻量图谱库（如 dagre、elkjs 或 react-flow），支持拖拽、缩放、节点高亮。
  - 交互：点击节点跳转到对应块；支持过滤只显示某类块或某条链路。
- 风险与建议：
  - 大图谱性能：采用分页/分层渲染与虚拟化；提供“仅显示最近 N 条链路”的开关。
  - 语义：为块类型与链接关系定义清晰的样式与标签。

### 6. 高级功能拓展（按优先级）

#### 6.1 角色图片差分管理
- 功能：支持角色立绘的差分管理（表情、服装、姿态），提供上传、裁剪、标签与批量导入。
- 实现思路：
  - 数据结构：Character 与 CharacterImage，支持变体类型与标签；在编辑器中提供快速切换。
  - 图片处理：内置基础裁剪与滤镜；支持拖拽上传与 URL 引入。
- 风险与建议：
  - 存储：图片与缩略图分离存储，避免阻塞主文档；提供缓存与懒加载。
  - 性能：图片压缩与懒加载，避免大文档渲染卡顿。

## 性能与可扩展性考量
- 输入性能：对 Markdown 解析设置延迟阈值（如 300ms），避免输入卡顿；对渲染态进行节流。
- 大文档优化：分块渲染、懒加载、虚拟列表；图谱视图采用分层/分页策略。
- 存储与同步：本地 IndexedDB 为主，可选云端同步；提供增量备份与冲突合并策略。
- 安全：严格过滤渲染 HTML，禁用 nodeIntegration，外部链接统一走浏览器。

## 故障排查与迁移建议
- 常见问题
  - 导入失败：检查 JSON 结构是否包含 blocks 数组；捕获并提示错误信息。
  - 协同编辑冲突：检查 yjs 版本与 tiptap collaboration 配置；确认共享协议可用。
  - 图谱卡顿：减少节点数量或启用“仅最近链路”；优化样式与布局算法。
- 迁移建议
  - 从桌面端迁移到移动端：遵循“核心抽取 + 新项目”策略，复制核心逻辑（EditorCore、BlockManager 等），仅实现移动端适配器。
  - 云同步：先实现本地 IndexedDB，再接入云服务；确保数据结构与接口稳定。
