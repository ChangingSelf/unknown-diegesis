# 小说块编辑器（Markdown+块编辑+双链）开发方案总结

## 核心目标
实现兼具「块编辑器灵活性」（拖拽/块管理）、「Markdown 即时渲染」（类似 Typora/Obsidian 原地预览）、「双链功能」（块间双向链接）的桌面端小说编辑器，本地优先存储，后期可扩展云端同步。


## 技术栈选型
| 模块                | 技术选择                          | 核心作用                                                                 |
|---------------------|-----------------------------------|--------------------------------------------------------------------------|
| 基础框架            | Electron + React + TypeScript    | 开发跨桌面应用（Windows/Mac/Linux），TypeScript 保障代码可维护性，React 适配编辑器生态 |
| 块编辑引擎          | tiptap                            | 管理块的创建/删除/拖拽/排序，处理编辑态光标与输入逻辑，支持自定义块类型（标题/正文/引用） |
| Markdown 解析渲染   | Lute（JS 版）                     | 将块内 Markdown 源码实时解析为 HTML 实现原地渲染，支持扩展 `[[双链]]` 语法 |
| 本地存储            | localForage + IndexedDB           | 存储块数据、小说元信息（章节/标题），支持大容量离线数据                     |
| UI 与样式           | Ant Design React + Tailwind CSS   | 提供基础组件（弹窗/菜单），快速实现响应式界面与自定义主题                  |
| 辅助工具            | DOMPurify（防 XSS）、docx（导出） | 过滤渲染 HTML 中的危险代码，支持导出 `.txt`/`.docx` 格式                  |


## 核心功能实现逻辑

### 1. 块数据结构（融合 Markdown 与双链）
```typescript
interface Block {
  id: string; // 块唯一标识（用于双链关联）
  type: 'heading' | 'paragraph' | 'quote'; // 块类型（映射 Markdown 元素）
  content: string; // 存储 Markdown 源码（如 "# 标题" 或 "正文 [[链接]]"）
  references: string[]; // 引用的其他块 ID（正向双链）
  referencedBy: string[]; // 被引用的块 ID（反向双链）
}
```


### 2. 「编辑态-渲染态」无缝切换（核心交互）
- **渲染态**：块显示 Lute 解析后的 HTML（如 `**加粗**` → 粗体文本），点击后切换为编辑态。
- **编辑态**：块显示 Markdown 源码，支持直接编辑，失焦（或按 `ESC`）自动切回渲染态。
- 实现：基于 tiptap 自定义块组件，通过 `isEditing` 状态控制显示模式，用 Lute 实时解析 `content` 字段。


### 3. 双链功能（基于 Markdown 语法扩展）
- **语法**：采用 Obsidian 式 `[[目标块ID]]` 标记双链。
- **创建**：输入 `[[` 触发块列表弹窗，选择目标块后自动补全 `[[目标块ID]]` 到 `content` 字段。
- **渲染**：Lute 解析时拦截 `[[...]]` 语法，替换为可点击链接（显示目标块标题），点击跳转至对应块。
- **关联维护**：新增/删除双链时，同步更新源块 `references` 和目标块 `referencedBy` 数组。


### 4. 块操作与 Markdown 兼容
- **拖拽排序**：通过 tiptap 拖拽 API 调整块顺序，仅修改章节内块 ID 列表，不改变块本身 `content`。
- **块类型映射**：Markdown 语法自动关联块类型（如 `# 标题` → 标题块），避免手动切换。
- **导出兼容**：基于 `content` 字段的 Markdown 源码，直接导出为纯 Markdown 文件或转换为 `.docx`。


## 开发优先级
1. **阶段1（2周）**：搭建 Electron + React + tiptap 框架，实现基础块编辑（增删改/拖拽）和本地存储。
2. **阶段2（2周）**：集成 Lute 实现 Markdown 即时渲染，开发编辑-渲染切换交互。
3. **阶段3（1-2周）**：实现双链语法解析、链接创建/跳转、关联维护，开发右侧引用面板。
4. **阶段4（1周）**：完善导出功能、快捷键、主题样式，优化大文档性能（限制实时渲染频率）。


## 后期扩展功能规划

### 1. 角色图片差分管理
**功能描述**：支持角色立绘的差分管理，包括不同表情、服装、姿态的图片切换

- **数据结构扩展**：

  ```typescript
  }

  }
  ```
- **图片管理器**：提供图片上传、裁剪、标签管理功能
- **实时预览**：在编辑器中快速切换角色图片差分
- **批量导入**：支持批量导入角色图片并自动分类

### 2. 图文混排
**功能描述**：支持图片与文字的混合排版，实现更丰富的视觉表现
**实现方案**：
- **扩展块类型**：添加 `image`、`mixed-content` 块类型
- **布局引擎**：基于 CSS Flexbox/Grid 实现响应式布局
- **图片嵌入**：支持拖拽上传、URL 引入、剪贴板粘贴
- **图文对齐**：提供左对齐、右对齐、居中、环绕等排版选项
- **图片处理**：内置基础图片编辑功能（裁剪、缩放、滤镜）

### 3. 富文本支持
**功能描述**：在 Markdown 基础上扩展更丰富的文本格式
**实现方案**：
- **扩展语法**：
  - 文本颜色：`{color:red}红色文本{/color}`
  - 字体大小：`{size:18px}大字体{/size}`
  - 自定义样式：`{style:font-family:楷体}楷体文本{/style}`
- **渲染扩展**：扩展 Lute 解析器支持自定义语法
- **格式工具栏**：提供可视化格式编辑工具
- **样式模板**：预设常用文本样式模板

### 4. 骰子支持
**功能描述**：为跑团类小说提供骰子功能，支持各种投掷规则
**实现方案**：
- **骰子语法**：
  - 基础投掷：`1d20`、`3d6`
  - 修正值：`1d20+5`、`2d6-2`
  - 特殊规则：`1d20k1`（取最高）、`4d6kh3`（保留最高3个）
- **实时计算**：输入时实时显示计算结果
- **历史记录**：保存投掷历史，支持重新计算
- **自定义规则**：支持用户自定义骰子规则和宏
- **可视化骰子**：提供 3D 骰子动画效果

## 关键注意事项
- **性能优化**：大文档场景下，设置 300ms 延迟解析 Markdown，避免输入时卡顿。
- **XSS 防护**：用 DOMPurify 过滤 Lute 生成的 HTML，防止恶意代码注入。
- **后期扩展**：核心逻辑（块处理/双链解析）抽离为独立工具库，预留云端同步接口和移动端适配层。双链后期开发，预留数据结构例如块id和块引用字段等方便后续扩展的内容

### 移动端适配预留架构

**1. 项目架构选择分析**
考虑到移动端开发可能在很久以后，**现阶段不建议采用 Monorepo**，原因如下：

**不适合 Monorepo 的原因：**
- **维护成本高**：Monorepo 需要复杂的构建配置和依赖管理
- **开发复杂度**：单项目开发阶段引入不必要的架构复杂性
- **学习成本**：团队需要学习 lerna、nx 等 Monorepo 工具
- **过度设计**：为远期功能增加当前开发的负担

**3. 核心逻辑复用层**
```typescript
// src/core/EditorCore.ts - 核心编辑器逻辑
export class EditorCore {
  constructor(
    private storageAdapter: StorageAdapter,
    private uiAdapter: UIAdapter,
    private interactionAdapter: InteractionAdapter
  ) {}

  // 平台无关的核心功能
  async createBlock(type: BlockType): Promise<Block> {
    const block: Block = {
      id: generateId(),
      type,
      content: '',
      references: [],
      referencedBy: [],
      metadata: { created: new Date() }
    };

    await this.storageAdapter.save(`block:${block.id}`, block);
    return block;
  }

  async updateBlock(id: string, content: string): Promise<void> {
    const block = await this.storageAdapter.load(`block:${id}`);
    block.content = content;
    block.metadata.modified = new Date();
    await this.storageAdapter.save(`block:${id}`, block);
  }

  parseMarkdown(content: string): ParsedContent {
    // 核心Markdown解析逻辑
    return markdownParser.parse(content);
  }
}

// src/core/BlockManager.ts - 块管理逻辑
export class BlockManager {
  private blocks: Map<string, Block> = new Map();

  addBlock(block: Block): void {
    this.blocks.set(block.id, block);
  }

  getBlock(id: string): Block | undefined {
    return this.blocks.get(id);
  }

  // 这些核心逻辑在移动端可以直接复用
}
```

**4. 实际移动端适配方案**
当需要开发移动端时，采用 **核心提取 + 新项目** 策略：

**方案对比：**

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| Monorepo | 统一管理，同步开发 | 复杂度高，维护成本大 | 大型团队，多平台并行开发 |
| 核心提取 + 新项目 | 简单直接，成本低 | 需要手动同步核心代码 | 单一项目，远期扩展 |
| 完全复制 | 最简单 | 代码重复，维护困难 | 功能简单，很少变更 |

**推荐：核心提取 + 新项目**
