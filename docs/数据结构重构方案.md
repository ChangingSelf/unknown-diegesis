## 1. 重构背景

### 1.1 现存问题

1. **目录结构不合理**
   - `workspace/workspace/` 嵌套重复
   - `chapters/` 命名不准确（正文不只有章节）
   - 缺少媒体资源管理目录

2. **数据存储不可靠**
   - 全局配置使用 `localStorage`，易丢失
   - 不符合 Electron 应用的标准实践

3. **类型设计过度**
   - 正文区分了章节、番外、序言等类型
   - 实际上这些在程序层面无区别

4. **缺少必要的资料类型**
   - 缺少世界观设定
   - 缺少大纲文档

### 1.2 重构目标

- ✅ 优化目录结构，语义清晰
- ✅ 统一文件命名规范
- ✅ 简化类型定义
- ✅ 增强数据可靠性
- ✅ 提升扩展性

## 3. 文件命名规范

### 3.1 命名格式

```
{timestamp}_{type}_{uuid}.{ext}
```

**组成部分：**

- `timestamp`: 创建时间，格式 `YYYYMMDD_HHMMSS`
- `type`: 文档类型（可选，仅资料类型需要）
- `uuid`: 12位随机ID
- `ext`: 文件扩展名（`.ud` 或 `.json`）

### 3.2 命名示例

```bash
# 正文文档（不包含类型前缀）
20260307_143052_abc123def456.ud

# 资料文档（包含类型前缀）
20260307_143105_character_def456abc123.ud
20260307_143120_location_ghi789jkl012.ud
20260307_143135_worldview_mno345pqr678.ud

# 配置文件
workspace.json
story.json
materials.json
```

### 3.3 命名优势

- ✅ 可读性强：一眼看出创建时间
- ✅ 自然排序：按创建时间排列
- ✅ 唯一性保证：UUID 避免冲突
- ✅ 类型识别：资料文档可快速识别类型

## 4. 数据类型定义

### 4.1 核心类型

```typescript
// 文档分类（只有两种）
type DocumentCategory = 'story' | 'material';

// 资料类型（保持细分，有实际意义）
type MaterialType =
  | 'character' // 角色设定
  | 'location' // 地点设定
  | 'item' // 物品设定
  | 'worldview' // 世界观设定（新增）
  | 'outline' // 大纲（新增）
  | 'timeline' // 时间线
  | 'note'; // 笔记

// 资源类型
type AssetType =
  | 'image' // 图片
  | 'document' // 文档
  | 'audio' // 音频
  | 'video'; // 视频
```

### 4.2 文档元数据

```typescript
interface DocumentMeta {
  id: string;
  title: string;
  category: DocumentCategory;

  // 仅资料类型需要
  materialType?: MaterialType;

  // 逻辑分组（所有文档都可用）
  folder?: string; // 多级文件夹用 "/" 分隔，如 "第一卷/第一章"
  order: number; // 在当前文件夹内的排序

  // 统计信息
  wordCount: number;

  // 时间戳
  created: string; // ISO 8601
  modified: string; // ISO 8601

  // 文件路径
  path: string;
}
```

## 5. 逻辑文件夹设计

### 5.1 设计理念

**物理扁平化 + 逻辑分组**

- 物理上：所有文档在各自的类型目录下
- 逻辑上：通过元数据 `folder` 字段组织

### 5.2 实现方式

```typescript
// 文档元数据示例
{
  "id": "abc123",
  "title": "第一章 开端",
  "category": "story",
  "folder": "第一卷",        // 逻辑文件夹
  "order": 1,                // 排序
  "path": "story/20260307_143052_abc123.ud"
}

{
  "id": "def456",
  "title": "番外：角色的日常",
  "category": "story",
  "folder": "番外篇/角色篇",  // 多级文件夹
  "order": 1,
  "path": "story/20260307_143105_def456.ud"
}
```

### 5.3 UI 展示

用户看到的逻辑结构：

```
├── 序言
├── 📁 第一卷
│   ├── 第一章 开端
│   ├── 第二章 发展
│   └── 第三章 转折
├── 📁 第二卷
│   └── ...
└── 📁 番外篇
    └── 📁 角色篇
        └── 番外：角色的日常
```

### 5.4 优势

- ✅ 灵活：用户可自由组织结构
- ✅ 高效：移动章节只需改元数据，不移动文件
- ✅ 扩展：支持任意层级嵌套

### 6.2 索引更新策略

1. **自动更新**：文档保存时同步更新索引
2. **启动校验**：应用启动时检查索引完整性
3. **增量更新**：只更新变更的部分，不全量重建
4. **手动重建**：提供"重建索引"功能（修复损坏）

### 6.3 索引优势

- ✅ 支持排序：按文件夹、时间等快速排序
- ✅ 引用追踪：记录资源被哪些文档使用

## 7. 全局配置存储

### 7.1 当前问题

```typescript
// ❌ 当前方案：使用 localStorage
localStorage.setItem('unknown-diegesis-recent-workspaces', JSON.stringify(data));
```

**问题：**

- 清除浏览器数据会丢失
- 不符合 Electron 应用规范
- 无法跨机器同步

### 7.2 重构方案

使用 Electron 的 `app.getPath('userData')`：

```typescript
// electron/main.ts
import { app } from 'electron';
import { join } from 'path';

// 获取用户数据目录
const userDataPath = app.getPath('userData');
// Windows: C:\Users\<用户名>\AppData\Roaming\unknown-diegesis
// macOS: ~/Library/Application Support/unknown-diegesis
// Linux: ~/.config/unknown-diegesis

const configPath = join(userDataPath, 'config.json');
const recentWorkspacesPath = join(userDataPath, 'recent-workspaces.json');
```

### 7.3 存储位置

```
# Windows
C:\Users\<用户名>\AppData\Roaming\unknown-diegesis\
├── config.json                 # 应用配置
├── recent-workspaces.json      # 最近工作区
└── logs/                       # 日志（可选）

# macOS
~/Library/Application Support/unknown-diegesis/
├── config.json
├── recent-workspaces.json
└── logs/

# Linux
~/.config/unknown-diegesis/
├── config.json
├── recent-workspaces.json
└── logs/
```

### 7.4 实现代码

```typescript
// electron/main.ts

// 读取最近工作区
ipcMain.handle('config:getRecentWorkspaces', async () => {
  const filePath = join(app.getPath('userData'), 'recent-workspaces.json');
  if (!existsSync(filePath)) {
    return { success: true, data: [] };
  }
  const content = await readFile(filePath, 'utf-8');
  return { success: true, data: JSON.parse(content) };
});

// 保存最近工作区
ipcMain.handle('config:saveRecentWorkspaces', async (_, workspaces) => {
  const configDir = app.getPath('userData');
  if (!existsSync(configDir)) {
    await mkdir(configDir, { recursive: true });
  }
  const filePath = join(configDir, 'recent-workspaces.json');
  await writeFile(filePath, JSON.stringify(workspaces, null, 2), 'utf-8');
  return { success: true };
});
```

## 8. 服务层重构

### 8.1 提取公共基类

```typescript
// src/services/base/BaseDocumentService.ts
export abstract class BaseDocumentService<TMeta> {
  protected abstract getTypeDir(): string;
  protected abstract getFilePrefix(): string;

  // 公共 CRUD 方法
  async getAll(workspacePath: string): Promise<TMeta[]> { ... }
  async getById(workspacePath: string, id: string): Promise<DocumentData<TMeta> | null> { ... }
  async create(workspacePath: string, data: Partial<TMeta>): Promise<TMeta | null> { ... }
  async save(workspacePath: string, id: string, data: DocumentData<TMeta>): Promise<boolean> { ... }
  async delete(workspacePath: string, id: string): Promise<boolean> { ... }

  // 文件名生成
  protected generateFileName(): string {
    const timestamp = new Date().toISOString()
      .replace(/[-:]/g, '').replace('T', '_').slice(0, 15);
    const uuid = crypto.randomUUID().replace(/-/g, '').slice(0, 12);
    const prefix = this.getFilePrefix();
    return prefix ? `${timestamp}_${prefix}_${uuid}.ud` : `${timestamp}_${uuid}.ud`;
  }
}
```

### 8.3 资料服务

// src/services/MaterialService.ts
export class MaterialService extends BaseDocumentService<DocumentMeta> {
  protected getTypeDir() {
    return 'materials';
  }
  protected getFilePrefix() {
    return '';
  } // 文件名中的类型前缀在 create 时指定

  // 资料特有方法
  async getByType(workspacePath: string, type: MaterialType): Promise<DocumentMeta[]> {

    const uuid = crypto.randomUUID().replace(/-/g, '').slice(0, 12);
    return `${timestamp}_${type}_${uuid}.ud`;
  }
}
